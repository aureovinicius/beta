1. Fundamentos de banco de dados

Modelo relacional e chaves

Um banco de dados é uma coleção organizada e persistente de dados inter-relacionados, gerenciada por um software chamado SGBD (Sistema Gerenciador de Banco de Dados), responsável por armazenamento, acesso concorrente, segurança e integridade. O SGBD cria uma camada de abstração entre os dados físicos (arquivos em disco) e o usuário/aplicação, evitando a redundância descontrolada e a dependência dos programas em relação à estrutura física dos dados.

O modelo relacional, proposto por Edgar F. Codd (IBM, 1970), é o paradigma dominante e fundamenta-se na teoria dos conjuntos e na lógica de predicados. Nele, os dados são organizados em relações (tabelas bidimensionais), e os relacionamentos entre os dados são representados por meio de valores (chaves), e não por ponteiros físicos. Esse é o ponto que mais cai em prova: no modelo relacional, ligações entre tabelas se fazem por valores de chaves, não por endereços de memória.

Para o Auditor-Fiscal, dominar tabelas e chaves é a base para consultar dados de NF-e, EFD/SPED, cadastro de contribuintes e cruzamentos fiscais, pois toda a inteligência fiscal repousa sobre bases relacionais bem modeladas.

Terminologia: relação, tupla, atributo, domínio

Na linguagem formal do modelo relacional, uma tabela é uma relação; cada linha é uma tupla (registro); cada coluna é um atributo (campo); e o conjunto de valores permitidos para um atributo é o domínio (ex.: o domínio de um CPF são strings de 11 dígitos numéricos).

Dois conceitos derivados são o grau (cardinalidade de atributos = número de colunas) e a cardinalidade (número de tuplas = número de linhas). Cuidado com a pegadinha: grau é sobre colunas, cardinalidade é sobre linhas. Uma propriedade essencial é que, formalmente, uma relação é um conjunto de tuplas — logo não admite tuplas duplicadas e a ordem das linhas e colunas é irrelevante.

Chave primária (PK) e chave candidata

Uma superchave é qualquer conjunto de atributos que identifica unicamente uma tupla. Uma chave candidata é uma superchave mínima (sem atributos supérfluos). Dentre as chaves candidatas, o projetista escolhe uma como chave primária (PK — Primary Key); as demais tornam-se chaves alternativas.

A PK tem duas restrições inseparáveis: unicidade (não há valores repetidos) e integridade de entidade — nenhum atributo da PK pode ser NULL. Por isso, uma coluna que admite NULL nunca pode ser, sozinha, chave primária. Uma PK formada por mais de um atributo é chamada chave composta (ou concatenada).

Chave estrangeira (FK) e integridade referencial

A chave estrangeira (FK — Foreign Key) é um atributo (ou conjunto) de uma tabela que referencia a chave primária de outra (ou da mesma) tabela. Ela materializa os relacionamentos no modelo relacional e impõe a integridade referencial: todo valor de FK deve corresponder a um valor existente de PK na tabela referenciada, ou ser NULL (quando permitido).

A integridade referencial é mantida por ações de propagação em UPDATE/DELETE, definidas no CREATE TABLE: ON DELETE CASCADE (apaga as filhas), SET NULL (anula a FK), RESTRICT/NO ACTION (proíbe a operação se houver filhas). Pegadinha clássica: a FK pode aceitar valores NULL e pode haver valores repetidos de FK (relacionamento 1:N), o que NÃO ocorre com a PK.

Regras de integridade e o catálogo

O modelo relacional define duas integridades fundamentais: a integridade de entidade (PK não nula e única) e a integridade referencial (FK consistente). Há ainda a integridade de domínio (valores dentro do domínio/tipo, garantida por CHECK, NOT NULL, DEFAULT) e integridades definidas pelo usuário (regras de negócio).

As definições de tabelas, colunas, chaves e restrições ficam no dicionário de dados (catálogo/metadados), que é a 'tabela das tabelas' consultada pelo SGBD para validar operações.

💡 Exemplo aplicado

Considere o cadastro de contribuintes do ICMS. Tabela CONTRIBUINTE(insc_estadual PK, cnpj, razao_social, municipio) e tabela NFE(chave_acesso PK, insc_emitente FK -> CONTRIBUINTE, valor, data). Aqui insc_emitente é uma FK: toda NF-e deve apontar para um contribuinte cadastrado (integridade referencial). Se um auditor tentar excluir um contribuinte que já emitiu notas, uma restrição ON DELETE RESTRICT bloqueará a operação, preservando o histórico fiscal.

Questão típica (estilo FGV/Cebraspe): 'Em um banco relacional, uma chave estrangeira pode assumir valor nulo.' — CERTO, desde que a coluna permita NULL; isso representa um relacionamento opcional. Já a afirmação 'a chave primária pode conter valores nulos' é ERRADA, por violar a integridade de entidade.

🎯 Na prova

A pegadinha mais frequente troca as propriedades de PK e FK: lembre que só a FK admite NULL e repetição. Outra armadilha confunde grau (colunas) com cardinalidade (linhas). E desconfie de afirmações que dizem que o modelo relacional usa 'ponteiros' ou 'endereços' para relacionar tabelas — isso é característica de modelos de rede/hierárquico, não do relacional.

Pontos-chave

  • Modelo relacional liga tabelas por VALORES (chaves), não por ponteiros físicos.
  • Grau = nº de colunas (atributos); cardinalidade = nº de linhas (tuplas).
  • PK: única e NÃO nula (integridade de entidade). Chave candidata mínima vira PK.
  • FK referencia PK de outra tabela; garante integridade referencial.
  • FK PODE ser nula e PODE repetir; PK não pode nenhum dos dois.
  • Ações referenciais: CASCADE, SET NULL, RESTRICT/NO ACTION.
  • Relação é um conjunto: sem tuplas duplicadas e ordem das linhas irrelevante.
