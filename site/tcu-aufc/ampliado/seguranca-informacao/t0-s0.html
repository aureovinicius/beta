1. Gestão de identidade e acesso

Autenticação, autorização e SSO

Três conceitos costumam ser confundidos e merecem distinção precisa. Identificação é declarar quem se é (ex.: informar o login). Autenticação (authentication, AuthN) é o processo de provar que essa identidade é verdadeira, comprovando algo que o usuário sabe (senha), possui (token) ou é (biometria). Autorização (authorization, AuthZ) ocorre depois da autenticação e decide o que a identidade autenticada pode fazer — quais recursos pode acessar e com quais permissões. Uma quarta peça, a auditabilidade (accounting/accountability), registra o que foi feito.

O SSO (Single Sign-On) é um mecanismo que permite ao usuário autenticar-se uma única vez e acessar diversos sistemas/aplicações sem refazer login a cada um. O SSO não é um protocolo, mas um padrão de arquitetura implementado por protocolos como SAML, OAuth 2.0 e OpenID Connect (OIDC). É essencial separar o que cada protocolo resolve: SAML e OIDC tratam de autenticação federada (provar identidade), enquanto OAuth 2.0, em sua essência, trata de autorização delegada (conceder acesso a recursos sem entregar credenciais).

No contexto federado surgem dois papéis centrais: o Identity Provider (IdP), que autentica o usuário e emite asserções/tokens, e o Service Provider (SP) ou Relying Party (RP), que confia no IdP e consome essas asserções para liberar acesso.

Autenticação x autorização: a fronteira que a banca explora

Autenticação responde 'quem é você?' e autorização responde 'o que você pode fazer?'. A ordem importa: não se autoriza sem antes autenticar (salvo acesso anônimo deliberado). Os fatores de autenticação classificam-se em: conhecimento (something you know — senha, PIN), posse (something you have — token OTP, smartcard, smartphone) e inerência (something you are — biometria: digital, facial, íris). Fatores adicionais menos clássicos são localização (somewhere you are) e comportamento (something you do). Cebraspe gosta de afirmar que biometria, sozinha, é 'mais segura' — cuidado: biometria não é segredo (sua face/digital são expostas) e não pode ser trocada se vazar, sendo um fator, não uma panaceia.

SAML 2.0

O SAML (Security Assertion Markup Language) é um padrão OASIS baseado em XML para troca de asserções de autenticação e autorização entre IdP e SP, muito usado em ambientes corporativos e federações acadêmicas (ex.: CAFe da RNP). O fluxo típico é o SP-initiated com binding HTTP-Redirect/POST: o usuário tenta acessar o SP, é redirecionado ao IdP, autentica-se e o IdP devolve uma SAML Assertion assinada digitalmente (XML Signature), contendo a identidade e atributos. O SP valida a assinatura e libera o acesso. SAML viabiliza SSO web e é tipicamente associado a cenários enterprise/B2B. Ponto de atenção: SAML usa XML e assinatura XML; é mais 'pesado' que tokens JWT.

OAuth 2.0 e OpenID Connect

OAuth 2.0 (RFC 6749) é um framework de autorização delegada: permite que uma aplicação (cliente) obtenha acesso limitado a recursos do usuário hospedados em outro serviço, sem conhecer a senha do usuário, por meio de um access token emitido por um Authorization Server. Seus papéis são resource owner, client, authorization server e resource server. O fluxo recomendado para apps web/mobile modernos é o Authorization Code (com PKCE). OAuth 2.0 puro NÃO é protocolo de autenticação — não padroniza como provar a identidade do usuário. O OpenID Connect (OIDC) é a camada de identidade construída SOBRE o OAuth 2.0: adiciona o ID Token (um JWT assinado com claims sobre o usuário) e o endpoint UserInfo, padronizando a autenticação. Em síntese: use OAuth 2.0 para autorizar acesso a APIs; use OIDC quando precisar de login/SSO de usuário (o 'Login com Google/Microsoft').

Protocolos legados e tokens

Em redes Windows/Active Directory, o protocolo padrão de autenticação é o Kerberos (baseado em tickets emitidos por um KDC, com chave simétrica e TGT/TGS), que substituiu o NTLM. Para autorização e federação na web moderna, o token JWT (JSON Web Token, RFC 7519) é amplamente usado: é compacto, autocontido e assinado (JWS), podendo ser validado sem consultar o servidor (stateless). LDAP, por sua vez, é um protocolo de diretório (consulta a um repositório de identidades), e não, em si, um protocolo de SSO.

💡 Exemplo aplicado

Item estilo Cebraspe: 'O OAuth 2.0 é um protocolo de autenticação que comprova a identidade do usuário ao serviço, dispensando o uso do OpenID Connect.' — ERRADO. OAuth 2.0 é framework de AUTORIZAÇÃO (delegação de acesso via access token); a autenticação padronizada de identidade é provida pela camada OpenID Connect (ID Token). Em auditoria do TCU, ao avaliar um portal de serviços que integra login federado (ex.: gov.br, que usa OIDC), o auditor deve verificar se o ID Token é validado (assinatura, issuer, audience, expiração) e se o access token tem escopos mínimos.

🎯 Na prova

A pegadinha clássica é trocar os papéis: dizer que OAuth 2.0 'autentica' ou que SAML 'é o mesmo que OAuth'. Memorize: OAuth = autorização; OIDC e SAML = autenticação/federação; SSO = conceito de login único. Outra armadilha: afirmar que SSO elimina a necessidade de autenticação — falso; o usuário ainda se autentica uma vez (e a falha do IdP vira ponto único de falha).

Pontos-chave

  • Autenticação (AuthN) prova identidade; autorização (AuthZ) define permissões; ocorrem nessa ordem.
  • Fatores: conhecimento (senha), posse (token) e inerência (biometria).
  • SSO é arquitetura de login único, implementada por SAML, OAuth 2.0 e OIDC.
  • SAML: XML, assinatura XML, IdP/SP, comum em ambientes corporativos/federações.
  • OAuth 2.0 = autorização delegada (access token); NÃO autentica usuário por si só.
  • OpenID Connect = camada de autenticação sobre OAuth 2.0 (ID Token / JWT).
  • Kerberos autentica em redes AD; LDAP é diretório, não SSO.
