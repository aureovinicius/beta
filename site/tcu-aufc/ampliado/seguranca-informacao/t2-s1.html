3. Soluções de segurança

Controles para aplicações web

Aplicações web e Web Services são hoje a principal superfície de ataque dos órgãos públicos, pois ficam expostos à internet e processam dados sensíveis. A segurança de aplicações (AppSec) consiste em controles e testes que reduzem vulnerabilidades ao longo do ciclo de desenvolvimento (DevSecOps — 'shift left', segurança desde o início).

A referência mundial é a OWASP (Open Worldwide Application Security Project), entidade que mantém o OWASP Top 10 (as dez categorias de risco mais críticas em aplicações web) e diversos guias e padrões. Conhecer esses controles e os tipos de teste é essencial para a função de auditoria de TI.

OWASP Top 10 e principais riscos

O OWASP Top 10 é uma lista de conscientização das categorias de risco mais críticas. Em sua edição mais recente, destacam-se: Broken Access Control (controle de acesso quebrado — o risco nº 1), Cryptographic Failures (falhas de criptografia, antes 'exposição de dados sensíveis'), Injection (inclui SQLi e XSS), Insecure Design, Security Misconfiguration, Vulnerable and Outdated Components, Identification and Authentication Failures, Software and Data Integrity Failures, Security Logging and Monitoring Failures e Server-Side Request Forgery (SSRF). É uma lista de RISCOS/conscientização, não um padrão de certificação.

Controles essenciais de codificação segura

Validação e sanitização de entradas (nunca confiar no input do usuário); consultas parametrizadas / prepared statements (defesa contra SQLi); output encoding e Content Security Policy (CSP) contra XSS; controle de acesso aplicado no SERVIDOR (não confiar em verificações só no cliente); gestão segura de sessões (cookies com flags HttpOnly, Secure e SameSite; tokens anti-CSRF); criptografia de dados em trânsito (TLS) e em repouso; armazenamento de senhas com hashing forte e salt (ex.: bcrypt, Argon2 — nunca em texto claro nem com hash simples como MD5/SHA-1); cabeçalhos de segurança HTTP (HSTS, X-Content-Type-Options); e tratamento de erros sem vazar informações internas.

Tipos de teste de segurança

SAST (Static Application Security Testing): análise estática do código-fonte, sem executar a aplicação — detecta cedo, integra-se ao pipeline. DAST (Dynamic): testa a aplicação em execução, de fora (caixa-preta), simulando ataques. IAST combina ambos via instrumentação. SCA (Software Composition Analysis): identifica vulnerabilidades em bibliotecas/componentes de terceiros (dependências). Pentest (teste de intrusão): exploração manual/assistida que simula um atacante real. Esses testes integram o DevSecOps, automatizando a segurança no CI/CD.

Segurança de Web Services e APIs

Web Services (SOAP/REST) e APIs exigem controles específicos: autenticação e autorização robustas (OAuth 2.0/OIDC, API keys com escopo mínimo), validação rigorosa de payloads, rate limiting e throttling (anti-DoS/abuso), uso de TLS, e proteção por API Gateway e WAF. A OWASP mantém um Top 10 específico para APIs (API Security Top 10), no qual o Broken Object Level Authorization (BOLA) é um dos riscos mais frequentes — quando a API não verifica se o usuário tem direito ao objeto solicitado.

💡 Exemplo aplicado

Item comentado: 'A utilização de consultas parametrizadas (prepared statements) é um controle eficaz contra ataques de injeção de SQL.' — CERTO. Ao separar comando de dados, impede que a entrada do usuário seja interpretada como código SQL. Em auditoria do TCU sobre um sistema web, a equipe verifica: cobertura do OWASP Top 10, execução de SAST/DAST no pipeline, SCA para dependências vulneráveis, armazenamento de senhas com hashing forte e salt (achado crítico se forem MD5 ou texto claro) e uso de TLS. A ausência de testes de segurança e de gestão de vulnerabilidades configura fragilidade de controle.

🎯 Na prova

Cebraspe explora a diferença SAST x DAST: estático (código, sem executar) x dinâmico (app rodando). Outra pegadinha: tratar o OWASP Top 10 como 'norma de certificação' — é lista de conscientização de riscos. Cuidado com 'validar a entrada apenas no navegador é suficiente' — controles do lado cliente são burláveis; a validação e a autorização devem ocorrer no servidor. Hashing de senha exige salt; criptografia reversível para senha é erro.

Pontos-chave

  • OWASP Top 10 lista riscos de conscientização (não é certificação); Broken Access Control lidera.
  • Prepared statements defendem contra SQLi; output encoding/CSP contra XSS.
  • Controle de acesso deve ser aplicado no servidor, não apenas no cliente.
  • Senhas: hashing forte com salt (bcrypt/Argon2); nunca texto claro ou MD5/SHA-1.
  • SAST = código estático; DAST = app em execução; SCA = dependências de terceiros.
  • DevSecOps integra segurança ao CI/CD (shift left); pentest simula atacante real.
  • APIs exigem OAuth 2.0/OIDC, rate limiting, validação e API Gateway; BOLA é risco-chave.
