4. Armazenamento e backup

Backup e recuperação (RPO/RTO)

Backup é a cópia de segurança de dados para permitir sua recuperação após perda, corrupção, exclusão acidental, ataque (ransomware) ou desastre. Uma estratégia de backup eficaz define o que copiar, com que frequência, por quanto tempo reter, onde armazenar e como restaurar — sempre validada por testes de restauração. As métricas RPO e RTO dimensionam, respectivamente, a perda de dados tolerável e o tempo de recuperação aceitável.

Tecnologias como backup completo, incremental e diferencial, snapshots, deduplicação e ferramentas específicas (como o Oracle RMAN para bancos de dados) compõem o arsenal. A regra prática 3-2-1 (três cópias, em dois tipos de mídia, com uma off-site) é referência. Diferentemente do RAID, o backup protege contra perdas lógicas e desastres.

RPO e RTO

RPO (Recovery Point Objective) é a quantidade máxima de dados (medida em tempo) que a organização tolera perder em um incidente — define o ponto no tempo até o qual os dados precisam ser recuperáveis e, portanto, a frequência dos backups. Um RPO de 1 hora exige backups (ou replicação) ao menos a cada hora.

RTO (Recovery Time Objective) é o tempo máximo aceitável para restaurar o serviço após uma interrupção. Um RTO baixo exige mecanismos rápidos de recuperação (snapshots, replicação, sites em standby). RPO olha para trás (dados perdidos); RTO olha para frente (tempo de retomada). Ambos derivam da análise de impacto no negócio (BIA) e orientam o investimento em redundância.

Tipos de backup: completo, incremental e diferencial

O backup completo (full) copia todos os dados selecionados; é a base, mas consome mais tempo e espaço. O incremental copia apenas o que mudou desde o último backup (completo ou incremental), economizando espaço e tempo de cópia, porém a restauração requer o último full mais TODOS os incrementais subsequentes na ordem.

O diferencial copia o que mudou desde o último backup completo; cada diferencial cresce até o próximo full, mas a restauração precisa apenas do último full mais o último diferencial. Resumo: incremental é rápido de gravar e lento/complexo de restaurar; diferencial é o oposto. Essa diferença é cobrada com frequência. Esquemas de rotação (ex.: avô-pai-filho) e GFS organizam a retenção.

Snapshots e deduplicação

Snapshots capturam o estado de um volume ou VM em um instante, geralmente de forma instantânea via copy-on-write ou redirect-on-write, permitindo rollback rápido. São excelentes para recuperação pontual e janelas de backup curtas, mas atenção: um snapshot que reside no mesmo storage NÃO é backup — se o array falha, o snapshot se perde junto. Snapshot complementa, não substitui, o backup off-site.

A deduplicação elimina blocos de dados redundantes, armazenando apenas cópias únicas e referências, o que reduz drasticamente o espaço de armazenamento e o volume transmitido em backups e replicações. Pode ser na origem (source) ou no destino (target), em linha (inline) ou pós-processo. Combinada à compressão, otimiza retenção de longo prazo.

Backup de bancos de dados e Oracle RMAN

Bancos de dados exigem backup consistente (transacionalmente íntegro). O Oracle RMAN (Recovery Manager) é a ferramenta nativa de backup e recuperação do Oracle: faz backups completos e incrementais em nível de bloco, valida blocos corrompidos, gerencia retenção e integra-se ao catálogo de recuperação. Permite point-in-time recovery (PITR) usando os arquivos de redo/archive logs para recuperar o banco até um instante específico.

Backups podem ser a quente (online, com o banco aberto, exigindo modo ARCHIVELOG) ou a frio (offline, com o banco parado). O RMAN suporta backups incrementais com block change tracking (acelera identificando blocos alterados) e deduplicação/compressão. Distinguir backup físico (RMAN, datafiles) de lógico (Data Pump/expdp, objetos) é relevante.

💡 Exemplo aplicado

Item comentado: 'Para restaurar um sistema que utiliza estratégia de backup incremental, basta restaurar o último backup completo e o último backup incremental realizado.' — ERRADO. No incremental, a restauração exige o último backup completo MAIS todos os incrementais subsequentes, na ordem. A afirmação descreve o backup DIFERENCIAL, em que basta o último full e o último diferencial.

🎯 Na prova

A pegadinha mais cobrada é trocar a lógica de restauração de incremental e diferencial — memorize: incremental = full + TODOS os incrementais; diferencial = full + ÚLTIMO diferencial. Outra: afirmar que RPO é tempo de recuperação (é RTO) ou que snapshot/RAID substituem backup. Cuidado também com RMAN exigir ARCHIVELOG para backup a quente.

Pontos-chave

  • RPO = dados máximos perdidos (olha para trás, define frequência); RTO = tempo de retomada (olha para frente)
  • Incremental: rápido de gravar, restauração exige full + todos os incrementais
  • Diferencial: restauração exige só full + último diferencial
  • Snapshot é recuperação rápida, mas no mesmo storage NÃO é backup
  • Deduplicação remove blocos redundantes; reduz espaço e tráfego
  • Oracle RMAN faz backup/recuperação física, incremental por bloco e PITR (ARCHIVELOG)
  • Regra 3-2-1: 3 cópias, 2 mídias, 1 off-site; backup protege contra perda lógica e ransomware (RAID não)
