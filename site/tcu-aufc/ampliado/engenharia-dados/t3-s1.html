4. Fluxo de manipulação (ETL e pipelines)

Versionamento, logging e CI/CD

Pipelines de dados modernos são tratados como software ("data as code"): seu código, configurações e até a definição de transformações ficam sob controle de versão, são testados automaticamente e implantados por pipelines de integração e entrega contínuas (CI/CD). Isso traz rastreabilidade, reprodutibilidade e qualidade — atributos essenciais para auditoria.

Além disso, pipelines robustos exigem logging detalhado (registro de execução, erros e métricas), tolerância a falhas (retries automáticos e checkpoints para retomar de onde parou) e monitoramento. A combinação dessas práticas transforma scripts frágeis em fluxos de dados confiáveis, auditáveis e operacionalmente resilientes.

Versionamento de código e dados

O versionamento de código (Git) registra cada alteração nas transformações, esquemas e configurações do pipeline, permitindo identificar quem alterou o quê, quando e por quê — base de toda auditoria de mudanças. Práticas como branches, pull requests e revisão de código (code review) garantem governança sobre o que entra em produção.

No contexto de dados, surge também o versionamento de dados e de esquemas: ferramentas como DVC (Data Version Control) versionam datasets e modelos, e formatos de tabela como Delta Lake, Apache Iceberg e Apache Hudi oferecem "time travel" (consultar o estado dos dados em um instante passado) e controle de versão das tabelas no Data Lake. Migrações de esquema (ex.: Flyway, Liquibase) versionam a evolução da estrutura dos bancos.

Logging, observabilidade e tolerância a falhas

O logging registra eventos da execução do pipeline: início/fim de tarefas, contagem de registros processados, erros e tempos. Logs estruturados (JSON) alimentam ferramentas de observabilidade (ELK, Grafana, Prometheus) e são insumo direto para auditoria e troubleshooting. A observabilidade vai além: métricas, logs e traces permitem entender o comportamento do pipeline.

Tolerância a falhas é crítica: retries (novas tentativas automáticas com backoff) tratam falhas transitórias; checkpoints salvam o progresso para que, em caso de falha, o processamento retome do último ponto consistente em vez de recomeçar; e a idempotência garante que reexecutar uma etapa não duplique dados. Orquestradores como Apache Airflow (com DAGs), Dagster e Prefect coordenam tarefas, dependências, agendamento e tratamento de falhas.

CI/CD aplicado a pipelines de dados

Integração Contínua (CI) automatiza, a cada commit, a verificação do código: testes unitários das transformações, testes de qualidade de dados, validação de esquemas e análise estática. Entrega/Implantação Contínua (CD) automatiza a publicação das mudanças nos ambientes (dev, homologação, produção), reduzindo erros manuais.

Ferramentas como GitHub Actions, GitLab CI e Jenkins executam esses pipelines de automação. No mundo de dados, frameworks como dbt (data build tool) integram transformações SQL versionadas, testes de dados e documentação ao fluxo de CI/CD, materializando o conceito de "analytics engineering". Boas práticas incluem ambientes separados, infraestrutura como código (Terraform) e promoção controlada entre estágios.

Por que isso importa para auditoria

Para o controle externo, pipelines versionados e com logging completo significam trilha de auditoria: é possível reconstituir qual versão do código gerou determinado dado, quando e com quais parâmetros, garantindo reprodutibilidade e não repúdio. A separação de ambientes e a aprovação via pull request evidenciam segregação de funções e controle de mudanças (alinhados a COBIT 2019 e ITIL v4).

A ausência dessas práticas — pipelines manuais, sem versionamento, sem logs — é um achado de auditoria recorrente, pois compromete a confiabilidade, a integridade e a rastreabilidade das informações que apoiam decisões de gestão e fiscalização.

💡 Exemplo aplicado

Cenário de auditoria de TI: ao analisar um pipeline que consolida dados de despesas, o auditor solicita o histórico de versões (Git) das transformações e os logs de execução. Constata que uma carga falhou na madrugada, mas o pipeline retomou automaticamente via checkpoint e registrou o evento, sem duplicar registros (idempotência). Item de prova: "Em pipelines de dados, checkpoints permitem retomar o processamento a partir do último ponto consistente após uma falha, e a idempotência evita a duplicação de dados em reexecuções." — CERTO.

🎯 Na prova

Cebraspe pode contrastar Integração Contínua (verificação a cada commit) com Entrega/Implantação Contínua (deploy automatizado) — não confunda os termos. Cuidado com afirmações que tratam logging como opcional ou que dizem que retry e checkpoint são a mesma coisa: retry repete a tentativa; checkpoint salva o progresso para retomada. Idempotência cai como garantia contra duplicação.

Pontos-chave

  • Pipelines tratados como software: código sob Git, com branches, PRs e code review
  • Versionamento de dados/esquemas: DVC, Delta Lake/Iceberg/Hudi (time travel), Flyway/Liquibase
  • Logging estruturado alimenta observabilidade e serve de trilha de auditoria
  • Tolerância a falhas: retries, checkpoints e idempotência
  • Orquestração: Airflow (DAGs), Dagster, Prefect
  • CI = testes/validação automáticos a cada commit; CD = deploy automatizado entre ambientes
  • GitHub Actions/GitLab CI/Jenkins; dbt para transformações versionadas e testadas
  • Versionamento + logs garantem reprodutibilidade, rastreabilidade e segregação de funções
