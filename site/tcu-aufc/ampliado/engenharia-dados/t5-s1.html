6. Integração com nuvem

Armazenamento em nuvem e IA

O armazenamento de objetos em nuvem é a fundação dos Data Lakes e das arquiteturas de dados modernas. Serviços como Amazon S3 (AWS), Azure Blob Storage (Microsoft) e Google Cloud Storage (GCS) oferecem armazenamento praticamente ilimitado, durável, escalável e de baixo custo para dados estruturados, semiestruturados e não estruturados, com cobrança por uso.

Sobre essa base de armazenamento integram-se serviços de inteligência artificial e análise: os dados acumulados alimentam plataformas de machine learning e serviços de IA gerenciada (visão computacional, processamento de linguagem natural, IA generativa), formando o ciclo dado → modelo → insight que caracteriza as arquiteturas analíticas em nuvem.

Armazenamento de objetos: S3, Blob e GCS

O armazenamento de objetos guarda dados como objetos em contêineres planos (buckets no S3/GCS; containers no Azure Blob), cada objeto identificado por uma chave e acompanhado de metadados. Diferentemente do armazenamento de blocos ou de arquivos, é acessado via API/HTTP, escala horizontalmente de forma quase ilimitada e é otimizado para alta durabilidade (o S3 anuncia durabilidade de 99,999999999% — "11 noves") e disponibilidade.

Recursos comuns: classes de armazenamento por temperatura de acesso (quente, frio, arquivamento — ex.: S3 Glacier, Azure Archive) para otimizar custo; versionamento de objetos; políticas de ciclo de vida; criptografia em repouso e em trânsito; e controle de acesso por políticas e IAM. São a camada de armazenamento padrão de Data Lakes, onde convivem arquivos Parquet, JSON e logs.

Do armazenamento ao Data Lake e Lakehouse

Sobre o armazenamento de objetos constrói-se o Data Lake: repositório centralizado que retém dados brutos em seu formato nativo, sem esquema prévio (schema-on-read), barato e flexível. Para evitar que vire um "pântano de dados" (data swamp) sem governança, adicionam-se catálogo, metadados e formatos transacionais (Delta Lake, Iceberg, Hudi), originando o Lakehouse, que combina a economia do lake com as garantias do data warehouse.

Serviços de consulta como Amazon Athena, Google BigQuery e Azure Synapse permitem executar SQL diretamente sobre os arquivos no armazenamento de objetos, sem mover os dados para um banco dedicado, reforçando o desacoplamento entre armazenamento e processamento (compute).

Integração com IA e machine learning

Os dados armazenados em nuvem alimentam plataformas de ML gerenciadas — Amazon SageMaker, Azure Machine Learning, Google Vertex AI — que cobrem o ciclo de vida de modelos: preparação de dados, treinamento, ajuste, implantação e monitoramento. Há também serviços de IA prontos (cognitive/AI services) acessíveis por API para visão computacional, OCR, transcrição de fala, tradução e PLN, sem necessidade de treinar modelos do zero.

Mais recentemente, serviços de IA generativa e LLMs gerenciados (Amazon Bedrock, Azure OpenAI Service, Google Vertex AI) permitem incorporar geração de texto, sumarização e técnicas como RAG (Retrieval-Augmented Generation), que combinam um modelo de linguagem com a base de dados/conhecimento da organização para respostas contextualizadas. A engenharia de dados garante o pipeline que prepara, limpa e organiza os dados que sustentam esses modelos.

Segurança, custo e governança na nuvem

O modelo de responsabilidade compartilhada vale também aqui: o provedor protege a infraestrutura física; o cliente é responsável por configurar corretamente permissões (IAM/RBAC), criptografia, políticas de bucket e classificação de dados. Buckets públicos mal configurados são causa frequente de vazamento de dados — achado clássico de auditoria.

Governança de custos (lifecycle policies, classes de armazenamento adequadas), conformidade com a LGPD (localização e proteção de dados pessoais, inclusive em treinamento de IA), e rastreabilidade (logs de acesso, linhagem) são preocupações centrais. Para IA, somam-se temas de transparência, viés algorítmico e proteção de dados usados no treinamento.

💡 Exemplo aplicado

Cenário de auditoria de TI: o TCU avalia um Data Lake institucional em Amazon S3 que alimenta um modelo de detecção de irregularidades. O auditor verifica que um bucket está com acesso público indevido (risco de vazamento) e que faltam políticas de ciclo de vida, elevando custos. Item de prova: "No modelo de responsabilidade compartilhada, a configuração de permissões de acesso e a criptografia dos dados armazenados em serviços como o Amazon S3 são responsabilidade do cliente, não do provedor de nuvem." — CERTO.

🎯 Na prova

Cebraspe explora o armazenamento de objetos x blocos x arquivos — objetos são acessados por API e identificados por chave, não por hierarquia de diretórios tradicional. A pegadinha mais cobrada é a responsabilidade compartilhada: usar S3/Blob não transfere ao provedor a responsabilidade por configurar acessos e criptografia. Cuidado também ao confundir Data Lake (dados brutos, schema-on-read) com Data Warehouse (estruturado, schema-on-write).

Pontos-chave

  • S3, Azure Blob e GCS = armazenamento de objetos: buckets/containers, acesso via API, escala e durabilidade altas
  • Classes por temperatura (quente/frio/arquivo) otimizam custo; versionamento e lifecycle policies
  • Armazenamento de objetos é a base de Data Lakes (schema-on-read)
  • Lakehouse: lake + formatos transacionais (Delta/Iceberg/Hudi) + governança
  • Athena/BigQuery/Synapse: SQL direto sobre arquivos, desacoplando storage e compute
  • Plataformas de ML: SageMaker, Azure ML, Vertex AI; IA generativa: Bedrock, Azure OpenAI
  • RAG combina LLM com a base de dados da organização para respostas contextualizadas
  • Responsabilidade compartilhada: cliente configura IAM, criptografia e políticas (buckets públicos = risco)
