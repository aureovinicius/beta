1. Bancos de dados

Bancos NoSQL (MongoDB, Elasticsearch)

Bancos NoSQL ("Not Only SQL") surgiram para atender cenários em que o modelo relacional tradicional encontra limitações: volumes massivos de dados (Big Data), necessidade de escalabilidade horizontal, esquemas flexíveis e alta disponibilidade em sistemas distribuídos. Em vez de tabelas rígidas com esquema fixo e junções (joins), os bancos NoSQL adotam estruturas otimizadas para padrões de acesso específicos, frequentemente sacrificando parte das garantias ACID em favor de desempenho e escalabilidade.

Não existe um único "banco NoSQL": trata-se de uma família com quatro categorias principais — orientados a documentos (MongoDB, Couchbase), chave-valor (Redis, DynamoDB), famílias de colunas (Cassandra, HBase) e grafos (Neo4j). Sistemas de busca como o Elasticsearch, embora muitas vezes classificados como motores de busca/análise, também são tratados como bancos orientados a documentos com índice invertido. A escolha entre NoSQL e relacional depende do problema, não de "modernidade": cada modelo tem seu nicho.

As quatro categorias de NoSQL

Documento: armazena registros semiestruturados em JSON/BSON, onde cada documento pode ter campos próprios (esquema flexível). Ideal para catálogos, perfis e dados aninhados. Ex.: MongoDB.

Chave-valor: o mais simples; associa uma chave a um valor opaco. Extremamente rápido para cache e sessões. Ex.: Redis, Amazon DynamoDB.

Família de colunas (wide-column): armazena dados por colunas agrupadas, otimizado para escrita massiva e leitura de grandes volumes. Ex.: Apache Cassandra, HBase.

Grafo: modela entidades (nós) e relacionamentos (arestas) como cidadãos de primeira classe, ideal para redes sociais, detecção de fraude e análise de vínculos. Ex.: Neo4j.

MongoDB

MongoDB é o NoSQL de documentos mais popular. Armazena documentos em BSON (JSON binário), agrupados em coleções (equivalentes a tabelas). Não exige esquema fixo: documentos da mesma coleção podem ter estruturas diferentes. Suporta índices, agregações (aggregation pipeline) e sharding (particionamento horizontal por chave de shard) para escalabilidade, além de replica sets para alta disponibilidade.

A partir da versão 4.x, o MongoDB passou a oferecer transações multidocumento ACID, derrubando a ideia (frequente em provas antigas) de que NoSQL "nunca" tem transações. Ainda assim, a modelagem recomendada favorece a desnormalização e o aninhamento (embedding) de dados relacionados em um único documento, reduzindo a necessidade de junções.

Elasticsearch

Elasticsearch é um motor de busca e análise distribuído construído sobre a biblioteca Apache Lucene. Indexa documentos JSON usando um índice invertido, que mapeia termos para os documentos onde aparecem, permitindo busca textual (full-text search) rápida, relevância (scoring), busca por proximidade e agregações analíticas em tempo quase real.

É a peça central do stack ELK/Elastic Stack (Elasticsearch, Logstash, Kibana, Beats), amplamente usado para análise de logs, observabilidade, SIEM e busca em e-commerce. Organiza dados em índices distribuídos em shards e réplicas. Não é um substituto de banco transacional: seu foco é leitura, busca e análise, não consistência transacional forte.

Teorema CAP

O teorema CAP (Brewer) afirma que um sistema de dados distribuído não pode garantir simultaneamente as três propriedades: Consistência (Consistency — toda leitura recebe a escrita mais recente), Disponibilidade (Availability — toda requisição recebe resposta) e Tolerância a Partição (Partition tolerance — o sistema continua operando mesmo com falhas de comunicação na rede). Como partições de rede são inevitáveis em sistemas distribuídos, na prática a escolha real é entre consistência (CP) e disponibilidade (AP) quando ocorre uma partição.

Sistemas CP (ex.: MongoDB em configuração padrão, HBase) priorizam consistência, podendo recusar respostas durante uma partição. Sistemas AP (ex.: Cassandra, DynamoDB) priorizam disponibilidade, aceitando consistência eventual. O conceito BASE (Basically Available, Soft state, Eventual consistency) contrasta com o ACID dos relacionais: muitos NoSQL adotam consistência eventual, em que réplicas convergem para o mesmo estado após algum tempo.

💡 Exemplo aplicado

Item estilo Cebraspe: "Conforme o teorema CAP, um sistema de banco de dados distribuído pode garantir, ao mesmo tempo, consistência, disponibilidade e tolerância a partição." — ERRADO. O teorema estabelece que apenas duas das três propriedades podem ser garantidas simultaneamente quando há partição de rede; como a tolerância a partição é mandatória em sistemas distribuídos, a escolha prática recai entre consistência e disponibilidade. Em uma auditoria de TI, ao avaliar uma solução de catálogo de produtos com esquema variável e alto volume de leituras, o auditor pode considerar adequada a adoção de MongoDB; já para análise de logs de segurança e dashboards de observabilidade, o Elasticsearch seria a escolha técnica coerente.

🎯 Na prova

A pegadinha clássica é afirmar que o CAP permite as três propriedades juntas (sempre ERRADO) ou que NoSQL "não tem transações" (hoje desatualizado para o MongoDB). Cuidado também ao associar BASE a ACID: são opostos. Cebraspe gosta de testar o pareamento exemplo-categoria (ex.: Cassandra como AP, Neo4j como grafo, Redis como chave-valor).

Pontos-chave

  • NoSQL = família com 4 tipos: documento, chave-valor, coluna e grafo
  • MongoDB: documentos BSON, esquema flexível, sharding e replica sets; suporta transações ACID desde a v4.x
  • Elasticsearch: busca full-text via índice invertido (Lucene); base do Elastic Stack/ELK
  • Teorema CAP: só 2 de 3 (Consistência, Disponibilidade, Partição) sob partição de rede
  • Como partições são inevitáveis, a escolha real é CP vs AP
  • BASE (consistência eventual) contrasta com ACID dos relacionais
  • NoSQL não substitui relacional: cada um tem seu cenário ideal
