Autenticação e autorização (OAuth2, JWT, OpenID)

Autenticação (AuthN) é provar quem você é; autorização (AuthZ) é determinar o que você pode fazer. Em arquiteturas de APIs e microsserviços, esses controles são padronizados por OAuth 2.0 (autorização delegada), OpenID Connect (autenticação sobre OAuth2) e JWT (formato de token). Confundi-los é o erro mais comum — e o mais cobrado.

Esses protocolos sustentam o login único (SSO), o acesso a APIs de terceiros e a federação de identidades em governo (ex.: gov.br usa OpenID Connect/OAuth2). Para o auditor, são essenciais ao avaliar controles de acesso conforme ISO/IEC 27001 e boas práticas de segurança.

OAuth 2.0: autorização delegada

OAuth 2.0 (RFC 6749) é um framework de autorização — não de autenticação. Ele permite que uma aplicação (cliente) obtenha acesso limitado a recursos do usuário em outro serviço sem expor a senha do usuário. Papéis: Resource Owner (usuário dono dos dados), Client (aplicação que quer acesso), Authorization Server (emite tokens) e Resource Server (a API protegida).

O cliente obtém um access token (com escopos limitando o acesso) e o apresenta ao Resource Server. Fluxos (grant types): Authorization Code (apps web/server-side, o mais seguro — com PKCE para SPAs/mobile), Client Credentials (comunicação máquina-a-máquina, sem usuário), e os legados/desaconselhados Implicit e Resource Owner Password Credentials. Refresh tokens permitem renovar o access token (de vida curta) sem novo login.

OpenID Connect: autenticação sobre OAuth2

OpenID Connect (OIDC) é uma camada de identidade construída sobre o OAuth 2.0. Enquanto o OAuth2 cuida da autorização (acesso a recursos), o OIDC adiciona a autenticação padronizada: além do access token, o servidor emite um ID Token (sempre um JWT) com claims sobre a identidade do usuário (sub, name, email) e disponibiliza o endpoint UserInfo.

É a base de SSO moderno e de logins sociais ("entrar com Google"). O Identity Provider (IdP) autentica o usuário e emite a identidade. Não confundir com SAML, padrão mais antigo baseado em XML, comum em ambientes corporativos legados; OIDC é baseado em JSON/REST e mais adequado a mobile e SPAs.

JWT: formato de token

JSON Web Token (RFC 7519) é um formato compacto e autocontido de token, composto por três partes separadas por ponto e codificadas em Base64URL: Header (algoritmo e tipo), Payload (claims — informações como sub, exp, iss, aud) e Signature (assinatura). A assinatura (JWS) garante integridade e autenticidade, usando HMAC (chave simétrica, ex.: HS256) ou par de chaves assimétricas (ex.: RS256).

Pontos críticos de segurança: JWT assinado NÃO é criptografado — o payload é apenas codificado e legível por qualquer um; não coloque segredos no payload (use JWE se precisar confidencialidade). Sempre validar assinatura, expiração (exp), emissor (iss) e audiência (aud). Vulnerabilidade clássica: aceitar alg: none ou confundir algoritmos. Tokens são stateless (não exigem consulta a sessão no servidor), o que favorece escalabilidade, mas dificulta a revogação imediata (mitiga-se com expiração curta + refresh + listas de revogação).

Modelos de autorização

Para decidir o que um usuário autenticado pode fazer, usam-se modelos: RBAC (Role-Based Access Control — permissões atribuídas a papéis, e papéis a usuários); ABAC (Attribute-Based Access Control — decisões por atributos de usuário, recurso e contexto, mais granular); e o princípio do menor privilégio (least privilege). Em OAuth2, os scopes delimitam o que o token autoriza.

Boas práticas transversais: usar HTTPS sempre (tokens em trânsito), armazenar tokens com segurança (evitar localStorage para tokens sensíveis por risco de XSS; preferir cookies HttpOnly/SameSite), e separar claramente AuthN de AuthZ.

💡 Exemplo aplicado

Item de prova comentado: "O OAuth 2.0 é um protocolo de autenticação que verifica a identidade do usuário por meio de um ID Token assinado." — ERRADO. OAuth 2.0 é um framework de autorização, não de autenticação. Quem padroniza a autenticação e emite o ID Token é o OpenID Connect, construído sobre o OAuth2. Usar OAuth2 "puro" para autenticação é considerado antipadrão. Essa troca AuthN×AuthZ é a pegadinha mais frequente do tema.

🎯 Na prova

A pegadinha-mãe é trocar autenticação por autorização: grave OAuth2 = autorização e OpenID Connect = autenticação. Outra muito explorada: afirmar que JWT é criptografado (falso — é apenas assinado e codificado em Base64URL; o conteúdo é legível). Cebraspe também testa a natureza stateless do JWT (favorece escala, dificulta revogação) e pode confundir SAML (XML, legado) com OIDC (JSON/REST). Lembre que Base64 não é criptografia.

Pontos-chave

  • Autenticação (quem é) ≠ autorização (o que pode). OAuth2 = autorização; OIDC = autenticação.
  • OAuth 2.0 (RFC 6749): framework de autorização delegada; papéis Resource Owner, Client, Authorization Server, Resource Server.
  • Fluxo recomendado: Authorization Code com PKCE; Client Credentials para máquina-a-máquina.
  • OpenID Connect estende o OAuth2 e emite o ID Token (sempre JWT) com a identidade do usuário.
  • JWT (RFC 7519): Header.Payload.Signature em Base64URL; assinado garante integridade, mas NÃO é criptografado (payload legível).
  • Validar assinatura, exp, iss e aud; rejeitar alg: none.
  • JWT é stateless (escalável) mas de revogação difícil; usar expiração curta + refresh token.
  • Autorização efetiva via RBAC/ABAC, scopes e menor privilégio.
