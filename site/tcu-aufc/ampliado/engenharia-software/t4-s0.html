5. DevOps e integração contínua

Pipelines de CI/CD

CI/CD é a espinha dorsal das práticas DevOps. Integração Contínua (Continuous Integration — CI) é a prática de integrar e validar mudanças de código frequentemente, com build e testes automatizados a cada commit. Entrega Contínua (Continuous Delivery — CD) garante que o código validado esteja sempre pronto para implantação, com deploy acionado manualmente; já a Implantação Contínua (Continuous Deployment) automatiza também o deploy em produção.

O pipeline é a automação ponta a ponta desse fluxo (build, teste, análise, empacotamento, deploy). Para o auditor de TI, pipelines bem desenhados são controles de qualidade e segurança: garantem rastreabilidade, segregação de funções e ausência de alterações manuais não auditáveis em produção.

CI, CD e Continuous Deployment

CI: desenvolvedores integram seu trabalho ao repositório comum várias vezes ao dia; cada integração dispara um build automático e a bateria de testes, detectando conflitos e defeitos cedo (fail fast). Pressupõe controle de versão (Git) e uma branch principal sempre íntegra (trunk-based development é a abordagem mais alinhada).

Continuous Delivery: além de CI, o artefato validado é automaticamente preparado e fica pronto para implantar; a promoção a produção requer aprovação manual (gate). Continuous Deployment vai além: toda mudança que passa no pipeline é implantada automaticamente em produção, sem intervenção humana. A diferença central entre Delivery e Deployment é justamente a existência (ou não) de aprovação manual para o deploy em produção.

Estágios típicos de um pipeline

Um pipeline costuma ter estágios encadeados: (1) Source — gatilho por commit/push/PR; (2) Build — compilação e empacotamento (em Java, Maven/Gradle gerando JAR/WAR ou imagem de contêiner); (3) Test — testes unitários, de integração e de contrato; (4) Análise de qualidade e segurança — análise estática (SonarQube), SAST, varredura de dependências (SCA) e de imagens; (5) Package/Publish — publicação do artefato em um registry (ex.: Nexus, Artifactory, container registry); (6) Deploy — em ambientes de homologação e produção.

O princípio build once, deploy many: o mesmo artefato versionado e imutável percorre todos os ambientes, parametrizando apenas a configuração — evita o problema "funciona na minha máquina" e garante reprodutibilidade.

GitHub Actions e ferramentas

GitHub Actions automatiza pipelines diretamente no repositório via arquivos YAML em .github/workflows. Conceitos: workflow (o pipeline, disparado por events como push, pull_request, schedule), jobs (executados em runners, em paralelo ou sequência via needs), steps (passos de um job) e actions (componentes reutilizáveis do marketplace, ex.: actions/checkout, actions/setup-java). Segredos ficam em GitHub Secrets, nunca no código.

Outras ferramentas equivalentes: GitLab CI/CD (.gitlab-ci.yml), Jenkins (Jenkinsfile, pipeline as code), Azure DevOps Pipelines, CircleCI e Argo CD (para GitOps em Kubernetes). A tendência GitOps usa o repositório Git como fonte única da verdade do estado desejado da infraestrutura/deploy.

Estratégias de deploy e boas práticas

Para reduzir risco de implantação: Blue-Green (dois ambientes idênticos; troca-se o tráfego do azul para o verde, com rollback instantâneo); Canary (libera a nova versão para uma fração dos usuários e amplia gradualmente conforme métricas); Rolling update (substitui instâncias aos poucos); Feature flags (liga/desliga funcionalidades sem novo deploy).

Boas práticas e controles de auditoria: pipeline como código sob controle de versão; segregação de funções e aprovações para produção; gestão segura de segredos (cofres como HashiCorp Vault); rastreabilidade (cada deploy ligado a um commit e a uma execução de pipeline); ausência de acesso manual a produção. A esteira CI/CD é, ela própria, um alvo de ataque (supply chain) e deve ser protegida.

💡 Exemplo aplicado

Item de prova comentado: "Na entrega contínua (continuous delivery), toda alteração de código que passa nos testes automatizados é implantada automaticamente em produção, sem qualquer intervenção humana." — ERRADO. Essa descrição corresponde à implantação contínua (continuous deployment). Na entrega contínua (delivery), o artefato fica pronto para implantar, mas o deploy em produção depende de uma aprovação/gate manual. A diferença delivery×deployment está exatamente na automação (ou não) do passo final de produção.

🎯 Na prova

A distinção Delivery×Deployment é a pegadinha número um: delivery para na aprovação manual; deployment automatiza até produção. Cebraspe também confunde Blue-Green (dois ambientes, troca total de tráfego) com Canary (liberação gradual a uma fração de usuários). Lembre que CI exige controle de versão e branch principal sempre íntegra, e que segredos jamais devem ir no código (vão em cofres/secrets). Associe build once, deploy many à imutabilidade do artefato.

Pontos-chave

  • CI: integração frequente com build e testes automáticos a cada commit (fail fast); exige controle de versão.
  • Continuous Delivery: artefato sempre pronto para deploy, mas produção requer aprovação manual.
  • Continuous Deployment: deploy em produção totalmente automatizado, sem intervenção humana.
  • Estágios: source → build → test → análise (SonarQube/SAST/SCA) → package → deploy.
  • Build once, deploy many: o mesmo artefato imutável percorre todos os ambientes.
  • GitHub Actions: workflows em YAML com events, jobs, steps e actions; segredos em GitHub Secrets.
  • Estratégias de deploy de baixo risco: Blue-Green, Canary, Rolling e Feature flags.
  • Controles de auditoria: pipeline as code versionado, segregação de funções, rastreabilidade e segurança da esteira (supply chain).
