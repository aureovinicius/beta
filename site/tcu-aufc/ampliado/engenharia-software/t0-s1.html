1. Arquitetura de software

Padrões de integração (API Gateway, Service Mesh, CQRS)

Sistemas distribuídos exigem padrões para resolver problemas recorrentes de comunicação, roteamento, segurança e escalabilidade entre serviços. Três padrões frequentemente cobrados são o API Gateway (porta de entrada única), o Service Mesh (camada de infraestrutura para comunicação serviço-a-serviço) e o CQRS (separação de leitura e escrita).

Esses padrões são complementares e operam em planos diferentes: o API Gateway trata o tráfego norte-sul (cliente externo para o sistema), o Service Mesh trata o tráfego leste-oeste (entre serviços internos) e o CQRS é um padrão de modelagem de dados/fluxo. Conhecê-los é essencial para auditar arquiteturas modernas de governo digital.

API Gateway

O API Gateway é um ponto de entrada único que fica entre os clientes externos e os microsserviços internos. Ele recebe todas as requisições e as encaminha (roteamento) aos serviços apropriados, podendo agregar respostas de vários serviços em uma só (agregação/composição), e centraliza preocupações transversais (cross-cutting concerns): autenticação/autorização, rate limiting (limitação de taxa), throttling, cache, TLS termination, logging e transformação de protocolo.

Benefícios: desacopla o cliente da topologia interna, evita que o cliente conheça muitos endpoints e centraliza políticas de segurança. Risco: pode virar um gargalo ou ponto único de falha se mal dimensionado. Exemplos: Kong, AWS API Gateway, NGINX, Spring Cloud Gateway. Variante: padrão BFF (Backend for Frontend), um gateway por tipo de cliente (web, mobile).

Service Mesh

O Service Mesh é uma camada de infraestrutura dedicada que gerencia a comunicação serviço-a-serviço (leste-oeste) de forma transparente para a aplicação. Em vez de cada serviço implementar lógica de rede (retries, timeouts, circuit breaker, mTLS, balanceamento), essa lógica é delegada a um proxy sidecar (geralmente Envoy) injetado ao lado de cada instância de serviço.

Arquitetura: plano de dados (os sidecars que interceptam o tráfego) e plano de controle (configura e coordena os sidecars). Recursos: criptografia mútua (mTLS) entre serviços, observabilidade (métricas, tracing distribuído), controle de tráfego (canary, blue-green), políticas de segurança e resiliência. Exemplos: Istio, Linkerd, Consul Connect. Diferença-chave: o API Gateway protege a borda; o Service Mesh governa o interior.

CQRS (Command Query Responsibility Segregation)

O CQRS separa as operações de escrita (commands, que alteram estado) das operações de leitura (queries, que apenas consultam) em modelos distintos. Em vez de um único modelo para ler e gravar, há um modelo de escrita otimizado para regras de negócio/consistência e um ou mais modelos de leitura otimizados para consultas (frequentemente desnormalizados).

Vantagens: cada lado escala e evolui independentemente; leituras rápidas com projeções específicas. Frequentemente combinado com Event Sourcing (o estado de escrita é o log de eventos, e os modelos de leitura são projeções atualizadas pelos eventos), o que introduz consistência eventual entre escrita e leitura. Custo: complexidade adicional — só se justifica em domínios com forte assimetria entre leitura e escrita ou alta carga. Aplicar CQRS em CRUDs simples é antipadrão (overengineering).

Como os padrões se combinam

Numa arquitetura típica: o cliente acessa o API Gateway (borda, tráfego norte-sul), que autentica e roteia para os microsserviços; entre os serviços, o Service Mesh cuida de mTLS, resiliência e observabilidade (tráfego leste-oeste); internamente, um serviço de alta carga pode usar CQRS para separar suas leituras e escritas. São camadas distintas e não excludentes.

Outros padrões correlatos cobrados: Circuit Breaker (interrompe chamadas a serviço falho para evitar cascata), Saga (transações distribuídas por compensação), Bulkhead (isolamento de recursos) e Sidecar (componente auxiliar acoplado).

💡 Exemplo aplicado

Cenário de auditoria: ao avaliar uma plataforma de microsserviços do governo, o auditor verifica que toda a comunicação entre serviços internos usa HTTP em texto claro, sem criptografia. Recomendação: adotar um Service Mesh (ex.: Istio) para impor mTLS automático entre serviços, atendendo ao princípio de defesa em profundidade e à segurança de dados em trânsito (alinhado à ISO/IEC 27001 e à LGPD). O API Gateway, sozinho, protege apenas a borda, não o tráfego leste-oeste.

🎯 Na prova

A pegadinha mais comum é confundir os planos de atuação: associe API Gateway a tráfego norte-sul (cliente↔sistema) e Service Mesh a tráfego leste-oeste (serviço↔serviço). Outra: afirmar que o CQRS exige obrigatoriamente Event Sourcing (falso — são padrões frequentemente combinados, mas independentes). Cebraspe também testa se o candidato sabe que o sidecar fica ao lado do serviço, tornando a lógica de rede transparente para a aplicação.

Pontos-chave

  • API Gateway: ponto de entrada único (tráfego norte-sul), faz roteamento, agregação, autenticação, rate limiting e cache.
  • Service Mesh: camada de infraestrutura para comunicação serviço-a-serviço (leste-oeste) via proxy sidecar (Envoy); oferece mTLS, resiliência e observabilidade.
  • Service Mesh tem plano de dados (sidecars) e plano de controle; exemplos: Istio, Linkerd.
  • CQRS separa modelo de escrita (commands) do de leitura (queries); muitas vezes com Event Sourcing e consistência eventual.
  • API Gateway protege a borda; Service Mesh governa o interior — são complementares, não substitutos.
  • Aplicar CQRS em CRUD simples é overengineering; ele se justifica com forte assimetria leitura/escrita.
  • Padrões correlatos: Circuit Breaker, Saga, Bulkhead e Sidecar.
