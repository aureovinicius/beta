2. Design e programação

Padrões de projeto (GoF e GRASP)

Padrões de projeto (design patterns) são soluções reutilizáveis e testadas para problemas recorrentes de projeto de software orientado a objetos. Não são código pronto, mas descrições estruturadas (nome, problema, solução, consequências) que formam um vocabulário comum entre desenvolvedores.

Os dois conjuntos mais cobrados são os 23 padrões GoF (Gang of Four — Gamma, Helm, Johnson e Vlissides, 1994), de natureza concreta e implementável, e os padrões GRASP (General Responsibility Assignment Software Patterns, de Craig Larman), de natureza mais conceitual, voltados à atribuição de responsabilidades e ao controle de coesão e acoplamento.

Padrões GoF: classificação

Os 23 padrões GoF dividem-se em três categorias:

Criacionais (5) — tratam da criação/instanciação de objetos, abstraindo o processo: Singleton (instância única), Factory Method, Abstract Factory, Builder e Prototype.

Estruturais (7) — tratam da composição de classes e objetos em estruturas maiores: Adapter (compatibiliza interfaces), Bridge, Composite (árvores parte-todo), Decorator (adiciona comportamento dinamicamente), Facade (interface simplificada para subsistema), Flyweight e Proxy (representante/substituto).

Comportamentais (11) — tratam da interação e distribuição de responsabilidades: Strategy (algoritmos intercambiáveis), Observer (notificação de mudanças — base do pub/sub), Command, Template Method, Iterator, State, Chain of Responsibility, Mediator, Memento, Visitor e Interpreter.

Padrões GoF essenciais

Singleton: garante uma única instância de uma classe e ponto de acesso global (ex.: configuração, pool de conexões). Cuidado: pode ocultar dependências e dificultar testes (antipadrão se abusado).

Factory Method / Abstract Factory: delegam a criação de objetos a subclasses/fábricas, desacoplando o cliente das classes concretas. Strategy: encapsula algoritmos intercambiáveis (ex.: diferentes formas de cálculo de imposto). Observer: define dependência um-para-muitos para notificação automática (base de sistemas reativos e EDA). Decorator: adiciona responsabilidades a um objeto dinamicamente, alternativa flexível à herança. Adapter vs Facade: Adapter converte uma interface existente em outra esperada; Facade cria uma interface nova e simplificada para um subsistema complexo.

Padrões GRASP

Os 9 padrões GRASP guiam a atribuição de responsabilidades:

Information Expert (atribuir responsabilidade a quem tem a informação necessária); Creator (quem deve criar objetos de outra classe); Controller (objeto que recebe e coordena eventos do sistema); Low Coupling (minimizar dependências entre classes); High Cohesion (manter responsabilidades focadas e relacionadas); Polymorphism (variar comportamento por tipo via polimorfismo, evitando condicionais); Pure Fabrication (criar classe artificial, não do domínio, para manter coesão/baixo acoplamento — ex.: classe de persistência); Indirection (introduzir intermediário para desacoplar); Protected Variations (proteger pontos de variação por trás de interfaces estáveis).

Coesão, acoplamento e princípios SOLID

Os GRASP formalizam dois conceitos centrais: alta coesão (cada módulo faz uma coisa bem definida) e baixo acoplamento (módulos dependem pouco uns dos outros) — metas universais de bom projeto. Sistemas com alta coesão e baixo acoplamento são mais fáceis de entender, testar e manter.

Relacionados, os princípios SOLID (Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion) complementam os padrões: por exemplo, Open/Closed (aberto para extensão, fechado para modificação) é viabilizado por Strategy e Template Method; Dependency Inversion sustenta a injeção de dependências.

💡 Exemplo aplicado

Item de prova comentado: "O padrão Adapter e o padrão Facade têm o mesmo propósito: simplificar a interface de um subsistema complexo." — ERRADO. O Facade fornece uma interface unificada e simplificada para um subsistema; o Adapter converte a interface de uma classe existente em outra interface esperada pelo cliente (compatibilização), sem necessariamente simplificar. Cebraspe adora trocar definições de padrões estruturais entre si.

🎯 Na prova

Cebraspe testa a classificação (criacional/estrutural/comportamental) e troca padrões parecidos: Adapter×Facade×Proxy×Decorator. Memorize que Singleton e Factory são criacionais e que Observer e Strategy são comportamentais. Nos GRASP, a pegadinha é confundir Information Expert (quem tem o dado tem a responsabilidade) com Creator. Lembre que GRASP é mais conceitual (responsabilidades/coesão/acoplamento) e GoF é mais concreto (estruturas de classes/objetos).

Pontos-chave

  • GoF: 23 padrões em 3 grupos — criacionais (5), estruturais (7) e comportamentais (11).
  • Criacionais incluem Singleton e Factory; estruturais incluem Adapter, Decorator, Facade, Proxy; comportamentais incluem Strategy e Observer.
  • GRASP (Larman): atribuição de responsabilidades — Information Expert, Creator, Controller, Low Coupling, High Cohesion, Polymorphism, Pure Fabrication, Indirection, Protected Variations.
  • Metas centrais: alta coesão e baixo acoplamento.
  • Adapter converte interface existente; Facade cria interface nova e simplificada.
  • Observer é a base de notificação pub/sub e de sistemas reativos.
  • SOLID complementa os padrões (Open/Closed via Strategy; Dependency Inversion via injeção de dependência).
