3. APIs e integrações

Design e versionamento de APIs REST

REST (Representational State Transfer) é um estilo arquitetural definido por Roy Fielding (2000) para sistemas distribuídos sobre HTTP. Uma API RESTful expõe recursos (substantivos, identificados por URIs) manipulados por verbos HTTP padronizados, com mensagens autodescritivas. O bom design de APIs afeta usabilidade, manutenibilidade e interoperabilidade — pilares da interoperabilidade de governo (Padrão ePING).

O versionamento é necessário porque APIs evoluem, mas clientes existentes não podem quebrar; gerenciá-lo bem é parte da governança de integrações que o auditor avalia.

Restrições do estilo REST

Fielding define seis restrições: (1) cliente-servidor (separação de responsabilidades); (2) stateless — o servidor não guarda estado de sessão entre requisições; cada requisição é autossuficiente; (3) cacheável — respostas indicam se podem ser cacheadas; (4) interface uniforme (o coração do REST); (5) sistema em camadas; (6) código sob demanda (opcional).

A interface uniforme inclui: identificação de recursos por URI, manipulação por representações (JSON, XML), mensagens autodescritivas e HATEOAS (Hypermedia as the Engine of Application State — respostas trazem links para próximas ações). O Modelo de Maturidade de Richardson classifica APIs do nível 0 (RPC sobre HTTP) ao nível 3 (HATEOAS). APIs que não atendem às restrições são REST-like, não plenamente RESTful.

Verbos HTTP, idempotência e segurança

Mapeamento usual: GET (ler), POST (criar), PUT (substituir/atualizar integralmente), PATCH (atualizar parcialmente), DELETE (remover). Dois conceitos cruciais:

Métodos seguros (safe): não alteram estado no servidor — GET, HEAD, OPTIONS. Métodos idempotentes: executar N vezes tem o mesmo efeito de executar uma vez — GET, PUT, DELETE, HEAD. POST não é idempotente (cada chamada cria um novo recurso); PATCH geralmente não é. A idempotência é fundamental para resiliência: clientes podem reenviar PUT/DELETE com segurança após timeout. Para tornar criações seguras a retries, usa-se idempotency key.

Boas práticas de design

Usar substantivos no plural para coleções (/usuarios, /usuarios/42), evitar verbos na URI (a ação está no método HTTP), aninhar recursos com moderação (/usuarios/42/pedidos), usar códigos de status HTTP corretos (2xx sucesso, 3xx redirecionamento, 4xx erro do cliente — 400, 401, 403, 404, 409, 422, 429 —, 5xx erro do servidor). Recursos transversais: paginação, filtros, ordenação e campos parciais via query string; HATEOAS quando aplicável; respostas de erro padronizadas (ex.: RFC 7807 / Problem Details).

Documentação: o padrão de fato é a OpenAPI Specification (antiga Swagger), que descreve a API em YAML/JSON, permitindo geração de documentação, clientes e validação. Contratos claros viabilizam testes de contrato e integração.

Estratégias de versionamento

Principais abordagens: (1) versionamento na URI — /v1/usuarios, /v2/usuarios (simples e visível, mas "polui" a URI e fere a pureza REST de que o recurso é único); (2) por cabeçalho — Accept: application/vnd.empresa.v2+json (content negotiation, mais limpo, porém menos visível); (3) por query string — ?version=2.

Diferencie mudanças retrocompatíveis (adicionar campo opcional, novo endpoint — não exigem nova versão) de breaking changes (remover/renomear campo, mudar tipo, alterar semântica — exigem nova versão). Boas práticas: manter versões antigas por período de transição com política de depreciação (deprecation) anunciada (cabeçalhos Deprecation/Sunset) e seguir, quando possível, o versionamento semântico (SemVer: MAJOR.MINOR.PATCH).

💡 Exemplo aplicado

Item de prova comentado: "Em uma API RESTful, o método POST é idempotente, pois múltiplas requisições idênticas produzem o mesmo resultado no servidor." — ERRADO. POST não é idempotente: cada chamada normalmente cria um novo recurso (e pode gerar duplicatas em retries). São idempotentes GET, PUT, DELETE e HEAD. Para mitigar duplicações em POST, usa-se uma chave de idempotência. Outra confusão: PUT (substitui o recurso inteiro) × PATCH (atualiza parcialmente).

🎯 Na prova

Idempotência e segurança de métodos HTTP são campeões de incidência — memorize: POST não é idempotente nem seguro; GET é seguro e idempotente; PUT/DELETE são idempotentes mas não seguros. Cebraspe afirma que REST é stateful (falso, é stateless) ou que HATEOAS é obrigatório em toda API REST (na prática é o nível mais alto de maturidade, raramente implementado). Cuidado também com PUT×PATCH e com a ideia errada de que versionar por URI é a única forma válida.

Pontos-chave

  • REST tem 6 restrições; stateless e interface uniforme são centrais. HATEOAS é o nível 3 de Richardson.
  • Stateless: o servidor não mantém estado de sessão; cada requisição é autossuficiente.
  • Métodos seguros: GET, HEAD, OPTIONS. Idempotentes: GET, PUT, DELETE, HEAD. POST não é idempotente.
  • PUT substitui o recurso inteiro; PATCH atualiza parcialmente.
  • Usar códigos HTTP corretos (2xx/3xx/4xx/5xx) e substantivos (não verbos) nas URIs.
  • OpenAPI/Swagger é o padrão para documentar contratos REST.
  • Versionamento: por URI (/v1), por cabeçalho (Accept) ou query string; distinga mudanças retrocompatíveis de breaking changes.
  • Política de depreciação com cabeçalhos Deprecation/Sunset gerencia a transição entre versões.
