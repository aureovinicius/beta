4. DevOps, CI/CD e IaC

Infraestrutura como código (Terraform)

Infraestrutura como Código (IaC) é a prática de provisionar e gerenciar recursos de infraestrutura (redes, VMs, bancos, políticas) por meio de arquivos de definição versionados, em vez de configuração manual via console. Isso traz reprodutibilidade, auditabilidade, controle de versão e automação — pilares da cultura DevOps e da operação confiável em nuvem.

O Terraform, da HashiCorp, é a ferramenta de IaC mais difundida por ser multicloud e declarativa. Junto com IaC, pipelines de CI/CD automatizam build, teste e entrega, fechando o ciclo de entrega contínua.

IaC: declarativo x imperativo, mutável x imutável

IaC pode ser declarativa (você descreve o ESTADO desejado e a ferramenta calcula como chegar lá — ex.: Terraform, CloudFormation) ou imperativa (você descreve os PASSOS — ex.: scripts). O modelo declarativo é idempotente: aplicar a mesma definição repetidamente converge ao mesmo estado, sem efeitos colaterais. IaC também viabiliza infraestrutura imutável: em vez de alterar servidores existentes (mutável, sujeito a 'configuration drift'), substitui-se a infraestrutura por uma nova versão a partir do código. Benefícios: elimina o 'drift', reduz erros manuais e permite rollback.

Terraform: linguagem, estado e fluxo

O Terraform usa a linguagem declarativa HCL (HashiCorp Configuration Language) e providers (plugins) para falar com cada nuvem (AWS, Azure, GCP) — por isso é multicloud/cloud-agnostic. Conceito central: o tfstate (arquivo de estado) mapeia os recursos descritos no código aos recursos reais provisionados; deve ser armazenado de forma remota e segura (ex.: backend S3 + DynamoDB para travamento), pois pode conter dados sensíveis. Fluxo de trabalho: terraform init (inicializa providers) → terraform plan (mostra o que será criado/alterado/destruído, sem aplicar) → terraform apply (aplica) → terraform destroy (remove). O plan é a etapa de revisão que antecede a mudança real.

Ferramentas correlatas e diferenças

CloudFormation (AWS) e Azure Resource Manager (ARM)/Bicep são as soluções nativas (single-cloud) de IaC declarativa. Ansible, Chef e Puppet são ferramentas de gestão de CONFIGURAÇÃO (instalam pacotes e configuram SO dentro das máquinas), enquanto Terraform foca em provisionar a INFRAESTRUTURA — muitas vezes se combinam. Pegadinha: Ansible é majoritariamente para configuração (e é agentless, via SSH); Terraform é para provisionamento de infraestrutura. Confundir os dois é erro comum.

CI/CD: integração e entrega contínuas

CI (Continuous Integration) integra e testa código com frequência, executando build e testes automatizados a cada commit. CD pode significar Continuous Delivery (a entrega até produção fica pronta, mas o deploy final é uma decisão/aprovação manual) ou Continuous Deployment (todo commit aprovado vai automaticamente a produção, sem intervenção humana). Ferramentas: Jenkins, GitHub Actions, GitLab CI, Azure DevOps Pipelines, AWS CodePipeline. Boas práticas incluem 'pipeline as code', testes automatizados, e estratégias de deploy de baixo risco (blue-green, canary, rolling). GitOps usa o Git como fonte única da verdade para infraestrutura e deploy.

💡 Exemplo aplicado

Cenário aplicado/auditoria: a equipe de TI do TCU provisiona todo o ambiente de um sistema via Terraform versionado em Git, com pipeline no GitHub Actions executando 'terraform plan' em pull requests e exigindo aprovação antes do 'apply'. O tfstate fica em backend S3 cifrado, com travamento via DynamoDB. Em auditoria, isso evidencia rastreabilidade e segregação de funções. Um achado negativo seria infraestrutura criada manualmente no console (sem IaC), gerando 'drift' e impossibilidade de reproduzir o ambiente para DR.

🎯 Na prova

Pegadinhas: (1) 'terraform plan' já aplica as mudanças — FALSO, ele apenas mostra o plano; (2) confundir Terraform (provisionamento, multicloud) com Ansible (configuração, agentless); (3) tratar Continuous Delivery e Continuous Deployment como sinônimos — a diferença é a aprovação manual antes da produção. Saiba que IaC declarativa é idempotente.

Pontos-chave

  • IaC versiona a infraestrutura; traz reprodutibilidade, auditabilidade e rollback
  • Declarativo (estado desejado, idempotente) x imperativo (passos); imutável combate o configuration drift
  • Terraform: HCL, multicloud via providers; tfstate mapeia código a recursos reais (proteger e remotizar)
  • Fluxo Terraform: init → plan (revisão, não aplica) → apply → destroy
  • Terraform provisiona infraestrutura; Ansible/Chef/Puppet gerenciam configuração (Ansible é agentless)
  • CI testa a cada commit; Continuous Delivery exige aprovação manual, Continuous Deployment é automático até produção
