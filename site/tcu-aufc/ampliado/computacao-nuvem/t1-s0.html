2. Plataformas e arquitetura de soluções

AWS, Azure e Google Cloud

Os três principais hyperscalers — Amazon Web Services (AWS), Microsoft Azure e Google Cloud Platform (GCP) — dominam o mercado de nuvem pública. Embora ofereçam capacidades equivalentes, usam nomenclaturas diferentes para serviços análogos, e a banca costura questões pedindo a correspondência entre eles.

Conhecer o mapeamento dos serviços fundamentais (computação, armazenamento, banco, rede, identidade e serverless) entre os três provedores é o que mais cai. Vale também entender o conceito de regiões e zonas de disponibilidade, comum a todos.

Regiões, zonas de disponibilidade e edge

Todos os hyperscalers organizam sua infraestrutura em Regiões (áreas geográficas independentes) compostas por múltiplas Availability Zones (AZs) — datacenters isolados em falhas, mas interligados por baixa latência. Distribuir cargas entre AZs garante alta disponibilidade; distribuir entre Regiões atende latência, soberania de dados e DR. Há ainda locais de borda (edge/PoPs) usados por CDNs (AWS CloudFront, Azure Front Door/CDN, Google Cloud CDN) para entregar conteúdo próximo ao usuário.

Computação e armazenamento (mapeamento)

Máquinas virtuais: AWS EC2, Azure Virtual Machines, GCP Compute Engine. Armazenamento de objetos: AWS S3, Azure Blob Storage, Google Cloud Storage. Armazenamento de blocos: AWS EBS, Azure Managed Disks, GCP Persistent Disk. Funções serverless: AWS Lambda, Azure Functions, Google Cloud Functions. Contêineres gerenciados (Kubernetes): AWS EKS, Azure AKS, Google GKE. Conhecer S3 como armazenamento de OBJETOS (não de blocos) é recorrente.

Bancos de dados e rede

Banco relacional gerenciado: AWS RDS/Aurora, Azure SQL Database, Google Cloud SQL/Spanner. NoSQL: AWS DynamoDB, Azure Cosmos DB, Google Firestore/Bigtable. Data warehouse: AWS Redshift, Azure Synapse Analytics, Google BigQuery. Rede virtual isolada: AWS VPC, Azure VNet, GCP VPC. DNS: AWS Route 53, Azure DNS, Google Cloud DNS. Conectividade dedicada: AWS Direct Connect, Azure ExpressRoute, Google Cloud Interconnect.

Identidade e governança

Gestão de identidade e acesso: AWS IAM, Azure usa Microsoft Entra ID (antigo Azure Active Directory) somado a RBAC e Azure AD, GCP usa Cloud IAM. Gestão de chaves: AWS KMS, Azure Key Vault, Google Cloud KMS. Monitoramento: AWS CloudWatch, Azure Monitor, Google Cloud Monitoring (antigo Stackdriver). Faturamento/governança em escala: AWS Organizations, Azure Management Groups, Google Cloud Resource Manager. Atenção ao rebranding Azure AD → Microsoft Entra ID, que já aparece em editais recentes.

💡 Exemplo aplicado

Item de prova: 'O Amazon S3, o Azure Blob Storage e o Google Cloud Storage são serviços de armazenamento de objetos adequados para guardar grandes volumes de dados não estruturados, como arquivos de mídia e backups.' — CERTO. Os três são serviços de object storage, acessados por API/HTTP, ideais para dados não estruturados, com alta durabilidade. Não devem ser confundidos com armazenamento de blocos (EBS/Managed Disks/Persistent Disk), usado como disco de VMs.

🎯 Na prova

A Cebraspe explora o mapeamento cruzado (ex.: associar Lambda à Azure, ou chamar S3 de banco de dados/armazenamento de blocos). Memorize as tríades. Cuidado com BigQuery (data warehouse, não banco transacional) e com a renomeação Azure AD → Microsoft Entra ID.

Pontos-chave

  • Hyperscalers: AWS, Azure e GCP — mesmos conceitos, nomes diferentes
  • VMs: EC2 / Azure VM / Compute Engine; object storage: S3 / Blob / Cloud Storage
  • Serverless: Lambda / Azure Functions / Cloud Functions; Kubernetes: EKS / AKS / GKE
  • Data warehouse: Redshift / Synapse / BigQuery; NoSQL: DynamoDB / Cosmos DB / Firestore
  • Conectividade dedicada: Direct Connect / ExpressRoute / Cloud Interconnect
  • Regiões contêm múltiplas Availability Zones; Azure AD foi renomeado para Microsoft Entra ID
